# D21 cross-walk — sc=1171 / F3 user-canary writers

**Date**: 2026-05-23
**Author**: tech-lead (cross-walk, doc-only)
**Inputs**: PR #404 (D21 DR-watchpoint, awaiting merge), PR #394 (Linux ground-truth), F3 saga history
**Status**: framing call — RECOMMENDS the next dispatch; no code

## 1. TL;DR

**Picked hypothesis: B (parent-stack collision against the userspace `[stack]` region).**

D21 caught two **legitimate musl push instructions** writing into VA `0x7ffffffee458` — the *parent's* userspace `[stack]` band, not the kernel-allocated isolated vfork-child stack VMA at `[0x7F00_..., 0x7F40_...)` (`STACK_ASLR_MIN`, `kernel/src/mm/vma.rs:393`). The musl code is fine; the collision says **the vfork-child is executing on the parent's userspace `[stack]`** — i.e. the `[VFORK-STACK]` substitution in `alloc_vfork_child_stack` (`kernel/src/proc/mod.rs:3192`) was not applied for this clone or was bypassed.

The fix is a **userspace-stack-layout audit** (which RSP the child actually runs on, plus whether the `new_stack != 0` gate at `subsys/linux/syscall.rs:2341` matched), NOT a kernel SSP-canary fix and NOT an ABI alignment fix.

## 2. D21 evidence recap

Per PR #404 over 3/3 KVM trials:

| Channel | Slot | VA | Fired | Site |
|---|---|---|---|---|
| `raw_offset` | 2 | `0x7ffffffee458` | Yes, 2/3 | `ld-musl + 0x15630` (push %rbx prologue) AND `ld-musl + 0x1efb7` (syscall return handler prologue) |
| `rbp_derived` | 1 | varies | **Never** | — |

CS=0x23 → CPL=3 user-mode; both writes legal pushes by musl. The corrupted slot is at the raw-offset canary location, not the RBP-derived chain.

VA `0x7ffffffee458` sits **0x11ba8 below `0x7fff_ffff_ffff`** — i.e. inside the kernel's user-stack high-VA band (`USER_STACK_TOP` ≈ `0x0000_7FFF_FFFF_F000`, see PT init in `kernel/src/proc/elf.rs`), 100% confirmed NOT inside `[STACK_ASLR_MIN, STACK_ASLR_MAX)` = `[0x7F00_..., 0x7F40_...)`.

## 3. Hypotheses analysed

### A — Stack alignment (SysV AMD64 §3.2.2 16-byte RSP)

**Status: weak.** Per SysV AMD64 ABI 1.0 §3.2.2, RSP must be 16-byte aligned at the moment a `call` is issued (so that on entry the callee sees RSP+8 16-byte aligned). `0x7ffffffee458 & 0xF == 0x8` → 8-byte aligned, which is **the post-call state**, exactly what the ABI requires inside the prologue. No alignment violation. **Rejected.**

### B — Parent-stack collision (recommended)

**Status: strong.** `alloc_vfork_child_stack` substitutes an isolated 64 KiB VMA so the child runs at `top - 8` somewhere inside `[STACK_ASLR_MIN, STACK_ASLR_MAX)`, *not* in the parent's `[stack]`. The D21 writer landed at `0x7fff_ffff_e458`, which is **the parent's `[stack]`** — clear evidence the substitution did not happen for this clone, or the child returned into a frame predating the substitution.

Two sub-shapes to confirm:

- **B1**: `new_stack == 0` at the `clone` site → `subsys/linux/syscall.rs:2341` gate is false → no substitution; child runs on whatever RSP it inherited (= parent RSP). Easy to verify: grep the `[VFORK]` line that `subsys/linux/syscall.rs:2306` emits for `new_stack=0`.
- **B2**: `alloc_vfork_child_stack` returned `None` → fell into the `WARN: failed to allocate isolated stack; using caller-supplied new_stack=…` path (line 2365) → child runs on the caller-supplied VA which is the parent stack.

The musl helper at `+0x15630` and `+0x1efb7` doing legitimate pushes ON the parent stack while libxul's SSP-canary slot lives in the same band is precisely the corruption described in `subsys/linux/syscall.rs:2320-2334` comments — the saga we already documented but evidently still happens in this run.

### C — musl path divergence (syscall numbers / vDSO)

**Status: weak.** Both captured RIPs are musl prologues (`push %rbx`, syscall-return handler) that exist identically on Linux per the published ABI; they would push to whatever RSP they were called with regardless of which kernel handles the syscall. The writers are normal; the RSP they pushed into is the divergence. **Rejected as primary**, retained as P3 backup.

### D — PT_GNU_STACK miscompute

**Status: weak.** PT_GNU_STACK controls **executability** of the stack region (per ELF gABI §5.2 — PT_GNU_STACK p_flags maps to W^X policy), not its VA. Our `[VFORK-STACK]` VMA is mapped `PROT_READ|PROT_WRITE|PAGE_NO_EXECUTE` (line 3312) which is the correct W^X-correct value regardless of PT_GNU_STACK. PT_GNU_STACK could conceivably cause a #PF inside the child once it ran on the isolated stack — but it would not move the writer to the parent stack. **Rejected.**

## 4. Recommended next dispatch

**Agent: `abi-compatibility-engineer`** (NOT `aether-kernel-engineer`).

**Why this specialist**: the divergence is in the Linux personality's vfork stack substitution gate, not in a native kernel primitive. The fix lives at the boundary between `subsys/linux/syscall.rs` `CLONE_VM` arm and `proc::alloc_vfork_child_stack`; the engineer needs to reason about glibc/musl `__clone` ABI conventions, the `new_stack` argument's semantics under POSIX `clone(2)`, and AstryxOS's substitution policy.

**Scope** (one sentence): determine whether B1 or B2 is the active sub-shape for the sc=1171 wedge by grepping the `[VFORK]` `new_stack=…` and `[VFORK-STACK]` lines in a fresh KVM trial, then propose a fix shape (either widen the substitution to `new_stack == 0` callers if B1, or diagnose the `find_free_stack_range` `None` return if B2).

**Exit criteria**: a doc that states "for the failing clone, `new_stack` was `<value>` and `alloc_vfork_child_stack` returned `<Some(child_rsp)|None>`" plus a one-paragraph fix-shape recommendation. No code, no PR — the next agent designs the patch.

**Soft cap**: 60 minutes. Burst to 90 if a fresh KVM trial is needed.

**Linux ground-truth GDB walk needed first?** No. PR #394 already established that Linux passes brk(0)/brk(curr+0x2000) cleanly with both glibc and musl. The hypothesis B framing does not need a re-walk because the divergence is purely on the kernel-side substitution policy — Linux has no analogous policy (it just runs the child on the caller-supplied stack), so the comparison would be uninformative. Re-walk only becomes useful if B is falsified.

## 5. What to NOT do

- **Do NOT dispatch an `aether-kernel-engineer` fix-it for a SysV alignment "fix"**. Hypothesis A is rejected; the write VA is correctly 8-byte aligned at call-prologue time.
- **Do NOT dispatch a `userspace-engineer` to patch musl** — the writers are legitimate. Per the project invariant, no upstream-binary edits.
- **Do NOT dispatch a "make the SSP canary stronger" fix-it** — the canary is doing its job (it detected real overwrite). Replacing or relocating it just hides the bug.
- **Do NOT widen the DR-watchpoint to RBP-derived** — it never fired in 3/3 trials, retaining it is fine for confirmation only, not for advancing the investigation.
- **Do NOT pivot to a different saga axis (F3 SSP, M1 substrate, sc=1233)** — D21 is the most specific evidence we have for sc=1171 in three saga iterations; abandoning it would restart the discipline-1 anti-pattern from `feedback_saga_diagnostic_discipline_2026_05_20.md`.
- **Do NOT extend the vfork window past `execve`** — the substitution is correct as designed; what's wrong is the gate, not the policy.
